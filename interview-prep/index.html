blue team questions

An IDS (Intrusion Detection System) passively monitors network traffic and generates alerts when suspicious activity is detected — it does not take action. An IPS (Intrusion Prevention System) sits inline with traffic and actively blocks threats in real time.

The key distinction: IDS observes, IPS intervenes. IDS is placed out-of-band (receives a copy of traffic via SPAN/TAP). IPS is deployed inline, which means a misconfiguration can block legitimate traffic — so tuning matters. In modern deployments, NGFW with IPS capabilities often replaces standalone IPS devices.

The Lockheed Martin Cyber Kill Chain describes 7 stages of an attack: Reconnaissance → Weaponization → Delivery → Exploitation → Installation → Command & Control (C2) → Actions on Objectives.

From a defender's perspective, breaking the chain as early as possible is the goal. Stopping delivery (e.g., email filtering) is easier than containing an attacker who has already reached C2. MITRE ATT&CK extends this model with granular tactics and techniques mapped to real-world threat groups.

NIST SP 800-61 defines four phases: Preparation → Detection & Analysis → Containment, Eradication & Recovery → Post-Incident Activity.

In practice: start with scope and severity — is this a false positive or confirmed incident? Then isolate affected systems before any remediation to preserve evidence and prevent lateral spread. Document everything with timestamps. After recovery, a lessons-learned review identifies gaps in detection and response playbooks.

Alert fatigue is one of the top SOC challenges. Tuning strategy: first, baseline normal — understand what your environment's "noisy" processes are. Then whitelisting known-good infrastructure (vulnerability scanners, backup jobs, monitoring tools) eliminates a significant chunk of false positives.

From there, tighten detection logic with conditions: correlate multiple events (failed login + successful login from different IP) rather than alerting on single events. Review each rule's signal-to-noise ratio quarterly. Severity thresholds matter — not every alert needs to be high priority.

Detection is reactive — rules and signatures fire on known bad behavior. Threat hunting is proactive — assuming a breach has occurred and searching for evidence of an attacker who is evading your existing controls.

A hunt starts with a hypothesis (e.g., "Is there an attacker living off the land using LOLBins?") and uses raw logs, endpoint telemetry, and network data to either confirm or disprove it. Successful hunts result in new detection rules, closing the gap for future automated coverage.

Each OSI layer has distinct attack surfaces. Layer 1 (Physical): hardware tampering, rogue devices. Layer 2 (Data Link): ARP spoofing, MAC flooding, VLAN hopping. Layer 3 (Network): IP spoofing, ICMP tunneling. Layer 4 (Transport): SYN floods, port scanning. Layers 5-6: session hijacking, SSL stripping. Layer 7 (Application): SQLi, XSS, SSRF — the richest attack surface.

Defense-in-depth means controls at multiple layers: firewalls (L3-4), WAFs (L7), network segmentation (L2), and endpoint detection (L7+). Understanding which layer an attack targets helps map it to the right control.
